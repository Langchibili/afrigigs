public async rankBids(jobContext, bidList) {
  const isGlobal = jobContext.scope === "FOR_EVERYONE";
  
  return bidList.sort((a, b) => {
    if (!isGlobal) {
      // Sort by proximity match first if restricted regionally
      const proximityA = a.user.native_last_city === jobContext.target_city ? 0 : 1;
      const proximityB = b.user.native_last_city === jobContext.target_city ? 0 : 1;
      if (proximityA !== proximityB) return proximityA - proximityB;
    }
    // Secondary sort fallbacks
    if (b.user.trust_score_yes !== a.user.trust_score_yes) {
      return b.user.trust_score_yes - a.user.trust_score_yes;
    }
    return b.user.profile_completion_skilled - a.user.profile_completion_skilled;
  });
}

# AfriGigs — Full Project Plan

**A localized, pan-African, mobile-first gig marketplace**

---

## 1. Executive Summary

AfriGigs connects independent workers ("gig-runners") with local and international job posters across African markets, using an escrow-backed payment system, dynamic per-country bidding rules, and privacy-safe proximity matching. This plan expands the original blueprint into a build-ready roadmap: architecture, design system, data model, workflows, team structure, timeline, and risk plan.

**Tech stack (confirmed):**
- Frontend: **Next.js** + **MUI (Material UI)** as the component library, **Framer Motion** for animation, **custom CSS Grid/Flexbox layout system** (MUI's `Grid`/`Grid2` is explicitly *not* used — see §4.3)
- Mobile shell: **React Native** WebView wrapper (native GPS + push notifications bridged via `postMessage`)
- Backend: **Strapi** (headless CMS as business-logic/API layer)
- Database: **MySQL**
- Payments: Paystack / Flutterwave / DPO Group aggregators
- Geocoding: Google Geocoding API (reverse-geocode only, never raw coordinates to other users)

## 4. Frontend Design System

### 4.1 Visual Direction
"Clean, modern, trustworthy" — the dark Pan-African palette from the blueprint is kept but refined for MUI theming and layered depth (glassy elevation, soft glow accents) rather than flat blocks.

**Palette (as MUI theme tokens):**
```js
palette: {
  mode: 'dark',
  primary:   { main: '#00BFA6', dark: '#009688', light: '#5BE0CB' }, // Unity Green, brightened for AA contrast
  secondary: { main: '#FFC107', dark: '#C79100', light: '#FFD54F' }, // Gold
  error:     { main: '#EF5350', dark: '#D32F2F' },                   // Alert Red
  background:{ default: '#121212', paper: '#1A1A1A' },
  divider:   'rgba(255,255,255,0.08)',
}
```

**Elevation strategy:** replace MUI's default flat `elevation` shadows with a custom shadow scale (soft, colored, low-spread) — e.g. cards get `box-shadow: 0 8px 24px rgba(0,0,0,0.45), 0 0 0 1px rgba(255,255,255,0.04)`, active/hovered cards add a faint primary-color glow (`0 0 20px rgba(0,191,166,0.15)`).

**Gradients:** used sparingly as accents, not backgrounds — CTA buttons use a 135° green→teal gradient; the "Verified" badge uses a gold gradient sheen; hero section uses a radial gradient vignette behind a mesh/grain texture at low opacity.

**Typography:** `Inter` or `General Sans` for UI, tighter tracking on headings, a numeric tabular-figure font (e.g. `Space Grotesk` or the same family's tabular variant) for wallet balances and bid amounts so numbers don't jitter on update.

### 4.2 Motion (Framer Motion)
- **Page transitions:** shared-layout fade/slide (`AnimatePresence` + `layoutId`) between `/gigs` list and `/gigs/[id]` detail — the gig card animates into the detail hero.
- **Micro-interactions:** button press = scale 0.97 + shadow compress; bid submission = success checkmark draws in via `pathLength` animation.
- **Skeleton → content:** staggered fade/slide-up (`staggerChildren`) as gig cards populate, avoiding layout pop.
- **Verification badge:** subtle looping shimmer (gradient position animation) on "Unverified" state to draw the eye without being obnoxious; stops and morphs to a solid check once verified.
- **Trust/Proximity sort changes:** cards use `layout` prop so re-sorting animates positions rather than jump-cutting.
- Respect `prefers-reduced-motion` — all animations wrapped with a reduced-motion fallback (opacity-only).

### 4.3 Custom Grid System (No MUI Grid)
Per requirement, MUI's `Grid`/`Grid2` is not used anywhere. Instead, a small custom layout primitive set is built once and reused:

```
components/layout/
  Stack.tsx      → thin wrapper over CSS flexbox (gap, direction, align, justify, wrap)
  GridBox.tsx    → thin wrapper over CSS Grid (columns, rows, gap, responsive column-count via props + MUI useMediaQuery)
  Container.tsx  → max-width + responsive padding shell
```

Example `GridBox` usage for the gig list:
```tsx
<GridBox columns={{ xs: 1, sm: 2, md: 3, lg: 4 }} gap={3}>
  {gigs.map(g => <GigCard key={g.id} {...g} />)}
</GridBox>
```
Internally this resolves to native `display: grid; grid-template-columns: repeat(n, 1fr); gap: ...` with breakpoints pulled from the MUI theme's `theme.breakpoints`, so spacing/typography stay token-consistent with MUI while layout math stays custom and fully controllable (e.g. asymmetric featured-gig layouts, masonry-style profile galleries) which MUI's Grid can't easily do.

- `GigCard` (compact/expanded variants), `TrustBadge`, `VerifiedTag`, `BidTicket`, `WalletBalanceChip`, `ProfileCompletionRing` (animated SVG progress), `CountryCurrencySelector`, `OnboardingStepper` (custom, not MUI `Stepper` visuals but same a11y pattern), `EscrowStatusTimeline`.

## 6. Core Workflows

### 6.1 Onboarding & Verification
`Register → Basic details + ID number + skills category → Instantly browse & bid (Unverified badge)` → optional `Upload scripted video selfie → Strapi review queue → Verified badge`.
- Unverified users can bid but their `BidTicket` renders with a caution chip; some employers may filter to verified-only.
- Verification review can start manual (admin panel in Strapi) and later add automated liveness/ID-match checks.

### 6.2 GPS & Proximity Matching
- Native app reads device GPS → sends raw lat/lng to WebView via a JS bridge message.
- Server reverse-geocodes to city-level only (`native_last_city`) — **raw coordinates are never persisted or exposed to other users**, only the resolved city string.
- `rankBids()` sorts: proximity (if job scope is `LOCAL`) → `trust_score_yes` → `profile_completion_skilled`, exactly as in the blueprint's TypeScript sort function — this ships as a Strapi custom controller.

### 6.3 Bidding Thresholds
- International bid (poster country ≠ bidder country): minimum **$5.00 USD** fixed floor.
- Domestic bid: minimum = `country.currency.minimum_national_bid`.
- Enforced in a Strapi lifecycle hook (`beforeCreate` on Bid) — errors surface as inline form validation in the Next.js `BidTicket` component, not raw 500s.

### 6.4 Escrow Ledger
`Poster funds job (Paystack/Flutterwave/DPO) → funds held in EscrowTransaction → worker delivers → poster confirms completion → funds move to Wallet.balance → worker withdraws to mobile money`.
- Add a **dispute path**: if poster doesn't confirm within N days of delivery marked, job auto-flags for admin review rather than silently staying locked.
- Add **partial release** support for milestone-based jobs (future phase).

## 10. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Currency/FX mismatches on international bids | Lock `proposed_amount_usd` at bid time using a snapshot rate; don't recompute live |
| Fake/GPS-spoofed verification videos | Manual review queue at launch; plan for liveness-detection vendor later |
| Escrow disputes with no resolution path | Build the auto-flag-to-admin timeout from day one, not as an afterthought |
| MUI + custom grid drift (inconsistent spacing) | Pull all spacing from `theme.spacing()`, never hardcode px, enforced via lint rule |
| Low-end Android performance with heavy animation | Gate non-essential Framer Motion effects behind a device-capability/`prefers-reduced-motion` check |
